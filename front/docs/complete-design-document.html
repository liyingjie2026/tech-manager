第一章 项目概述

1.1 项目背景

随着科技创新驱动发展战略的深入实施，科研项目管理工作日益复杂化、精细化。传统的科研项目管理方式存在信息孤岛、流程繁琐、监管困难等问题，严重制约了科研管理效率的提升。为了适应新时代科研管理的需求，提高科研项目管理的科学化、规范化和信息化水平，亟需建设一套功能完善、技术先进、安全可靠的科研项目管理系统。

本项目旨在构建一个集需求征集、项目申报、立项评审、过程管理、结题验收、成果管理于一体的综合性科研项目管理平台。系统采用先进的技术架构，支持多角色协同工作，实现科研项目全生命周期的数字化管理。

1.2 项目目标

本系统的建设目标是：

  • 建立统一的科研项目管理平台，实现需求征集、项目申报、评审立项、过程监管、结题验收、成果管理的全流程信息化管理。
  • 实现监测监管端、科研机构端、专家端三端协同，支持多角色在线办公，提高管理效率。
  • 建立完善的权限管理体系，实现页面按钮级的细粒度权限控制。
  • 构建灵活的流程引擎，支持可视化流程配置，适应不同类型项目的管理需求。
  • 采用国产化技术路线，基于麒麟操作系统和人大金仓数据库，确保系统安全可控。
  • 提供丰富的统计分析功能，为决策提供数据支撑。

1.3 建设原则

1.3.1 先进性原则

系统采用目前主流的前后端分离架构，前端基于Next.js 16框架，支持服务端渲染和静态生成，提供优秀的用户体验。后端基于Spring Boot 3.2框架，采用微服务架构，便于系统扩展和维护。数据库选用人大金仓KingbaseES，完全兼容PostgreSQL生态，性能稳定可靠。

1.3.2 实用性原则

系统功能设计紧密结合科研项目管理的实际需求，充分考虑各类用户的使用习惯，界面友好、操作简便。系统提供完善的帮助文档和操作指南，降低用户学习成本。

1.3.3 安全性原则

系统采用多层次的安全防护体系，包括网络安全、应用安全、数据安全等。支持多因素认证、数据加密传输、敏感信息脱敏、操作日志审计等安全机制，确保系统和数据安全。

1.3.4 可扩展性原则

系统采用模块化设计，各功能模块相对独立，便于后续功能扩展。系统预留接口，支持与其他业务系统集成。采用微服务架构，支持横向扩展，满足业务增长需求。

1.3.5 国产化原则

系统全面采用国产化技术栈，包括麒麟操作系统、人大金仓数据库、达梦数据库等，确保系统安全可控，符合国家信息安全要求。

第二章 需求分析

2.1 业务需求分析

2.1.1 需求征集管理

需求征集是科研项目管理的起点，系统需要提供完善的需求征集和管理功能。科研机构可以在线提交科研需求，管理部门对需求进行审核、分类和匹配，形成项目指南。

业务流程：
  1. 科研机构填写需求征集表，包括需求名称、所属领域、预期水平、技术成熟度等信息。
  2. 提交需求后，系统自动流转至监管部门审核。
  3. 监管部门审核需求，可选择通过、不通过或退回修改。
  4. 审核通过的需求纳入需求库，作为项目申报的依据。
  5. 系统支持需求匹配功能，将相似需求进行聚类分析。

2.1.2 项目申报管理

项目申报是科研项目立项的前置环节，系统需要支持在线申报、材料上传、查重检测等功能。

业务流程：
  1. 监管部门发布项目申报指南，明确申报要求和时间节点。
  2. 科研机构查看申报指南，在线填写申报书。
  3. 申报书包括项目基本信息、研究内容、技术路线、预算明细等。
  4. 上传附件材料，如可行性报告、单位资质等。
  5. 提交前进行查重检测，系统自动生成查重报告。
  6. 查重率符合要求后，正式提交申报材料。
  7. 系统记录申报时间，超过截止时间无法提交。

2.1.3 立项评审管理

立项评审是项目审批的核心环节，包括形式审查、专家评审、立项审批等步骤。

业务流程：
  1. 监管部门对申报材料进行形式审查，检查材料完整性和规范性。
  2. 形式审查通过后，进入专家评审环节。
  3. 系统从专家库中抽取评审专家，发送评审通知。
  4. 专家在线查看申报材料，填写评审意见和评分。
  5. 支持多轮评审，如初评、复评、答辩等。
  6. 评审结束后，系统自动汇总评审结果。
  7. 监管部门根据评审结果进行立项审批。
  8. 立项通过的项目进入任务书签订环节。

2.1.4 任务书管理

任务书是项目实施的依据，系统需要支持任务拆分、任务书生成、盖章上传、任务书审核等功能。

业务流程（分步骤）：
  1. 步骤1：拆分任务书 - 项目负责人在线拆分项目任务，包括任务名称、任务类型、负责单位、完成时间、考核指标等。
  2. 步骤2：导出任务书 - 系统根据任务拆分信息自动生成任务书Word文档，项目负责人下载任务书。
  3. 步骤3：上传已盖章任务书 - 项目负责人将盖章后的任务书扫描件上传至系统。
  4. 步骤4：提交审核 - 提交任务书至监管部门审核，审核通过后任务书生效。

2.1.5 项目变更管理

项目实施过程中可能需要变更项目内容、经费、时间等，系统需要支持变更申请和审批。

2.1.6 中期检查管理

中期检查是项目过程监管的重要手段，系统需要支持材料提交、专家评审、整改意见等功能。

2.1.7 年度检查管理

年度检查用于评估项目年度执行情况，系统需要支持年报提交、材料审核、检查评价等功能。

2.1.8 结题验收管理

结题验收是项目结束的标志，系统需要支持验收申请、验收评审、成果归档等功能。

2.1.9 成果管理

成果管理用于统一管理项目产生的科研成果，包括论文、专利、软件著作权等。

3.3 网络架构

3.3.1 网络拓扑

系统采用三层网络架构：接入层、应用层、数据层。接入层部署负载均衡器和防火墙，应用层部署Web服务器和应用服务器，数据层部署数据库和缓存服务器。各层之间通过VLAN隔离，提高安全性。

3.3.2 安全策略

  • 部署防火墙，限制外部访问，只开放80和443端口。
  • 启用SSL/TLS加密，所有HTTP请求强制跳转到HTTPS。
  • 应用层和数据层之间采用内网通信，禁止外部直接访问。
  • 部署入侵检测系统（IDS），实时监控网络流量。
  • 定期进行安全扫描和渗透测试。

3.4 数据架构

3.4.1 数据库设计原则

  • 规范化设计：遵循第三范式，减少数据冗余。
  • 性能优化：合理使用索引，优化查询性能。
  • 安全性：敏感数据加密存储，实现数据脱敏。
  • 可扩展性：预留扩展字段，便于后续功能扩展。

3.4.2 数据库架构

系统采用人大金仓KingbaseES作为主数据库，Redis作为缓存数据库。人大金仓支持主从复制，实现读写分离。Redis采用集群模式，提供高可用性。

数据库 类型 用途 部署模式
人大金仓 关系型数据库 业务数据存储 主从复制
Redis 缓存数据库 热点数据缓存、Session存储 集群模式

3.5 安全架构

3.5.1 安全层次

系统安全设计包括网络安全、应用安全、数据安全三个层次。

安全层次 安全措施 实现技术
网络安全 防火墙、入侵检测、DDoS防护 硬件防火墙、IDS/IPS
应用安全 身份认证、权限控制、数据验证 Spring Security、JWT、XSS防护
数据安全 数据加密、数据备份、数据脱敏 AES加密、定时备份、字段脱敏

3.5.2 身份认证机制

系统采用JWT（JSON Web Token）机制实现身份认证。用户登录成功后，服务器签发JWT令牌，客户端在后续请求中携带令牌。服务器验证令牌的有效性，完成身份认证。

身份认证流程
1. 用户输入用户名和密码
↓
2. 前端提交登录请求
↓
3. 后端验证用户名和密码
↓
4. 验证通过，签发JWT令牌
↓
5. 前端存储令牌到LocalStorage
↓
6. 后续请求携带令牌进行身份验证

3.5.3 权限控制机制

系统实现基于RBAC（Role-Based Access Control）的权限控制模型，支持页面按钮级的细粒度权限控制。

  • 角色管理：系统预定义超级管理员、监管管理员、科研机构管理员、专家等角色，管理员可自定义角色。
  • 权限管理：权限包括菜单权限、页面权限、按钮权限、数据权限。
  • 用户授权：为用户分配角色，用户继承角色的所有权限。
  • 权限验证：前端根据权限控制菜单和按钮显示，后端在接口层进行权限验证。

第五章 数据库设计

5.1 数据库设计原则

数据库设计遵循以下原则：

  • 规范化设计：遵循第三范式（3NF），减少数据冗余，提高数据一致性。
  • 性能优化：合理设计索引，优化查询性能。对于频繁查询的字段建立索引。
  • 安全性：敏感字段加密存储，如密码、身份证号等。
  • 可扩展性：预留扩展字段，便于后续功能扩展。
  • 数据完整性：使用外键约束、唯一约束、非空约束等保证数据完整性。

5.2 数据库表设计

系统共设计32张数据库表，分为以下几类：

  • 用户权限类（6张表）：用户表、角色表、权限表、用户角色关联表、角色权限关联表、机构表
  • 需求管理类（1张表）：需求表
  • 项目管理类（15张表）：项目表、项目申报表、项目评审表、专家评审表、任务书表、任务表、项目变更表、中期检查表、年度检查表、验收表、项目成员表、项目附件表等
  • 成果管理类（2张表）：成果表、成果附件表
  • 专家管理类（2张表）：专家表、专家领域表
  • 流程引擎类（4张表）：流程定义表、流程节点表、流程实例表、流程日志表
  • 系统管理类（2张表）：字典表、操作日志表

5.4 数据库适配说明

5.4.1 人大金仓数据库适配

人大金仓KingbaseES是一款国产关系型数据库，完全兼容PostgreSQL生态。系统数据库设计完全兼容人大金仓数据库，具体适配说明如下：

  • 数据类型：人大金仓支持PostgreSQL的所有数据类型，包括BIGINT、VARCHAR、TEXT、TIMESTAMP、DECIMAL等。
  • 索引：支持B-Tree、Hash、GiST、GIN等多种索引类型。
  • 约束：支持主键约束、外键约束、唯一约束、非空约束、检查约束等。
  • 函数：支持PostgreSQL的内置函数和自定义函数。
  • 事务：支持ACID事务特性，确保数据一致性。
  • 备份恢复：支持物理备份和逻辑备份，提供完善的备份恢复机制。

5.4.2 数据库连接配置

Spring Boot应用连接人大金仓数据库的配置示例：

spring:
  datasource:
    driver-class-name: com.kingbase8.Driver
    url: jdbc:kingbase8://localhost:54321/research_db
    username: system
    password: ******
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
注意事项：
  • 人大金仓默认端口为54321，与PostgreSQL的5432不同。
  • 驱动类名为com.kingbase8.Driver。
  • JDBC URL格式为jdbc:kingbase8://。
  • 需要在pom.xml中添加人大金仓JDBC驱动依赖。
